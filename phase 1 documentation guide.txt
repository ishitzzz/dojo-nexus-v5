Phase 1 Prompt — Conversational Onboarding

SYSTEM CONTEXT AND ROLE
You are a senior full-stack architect working on a personalized learning platform built with Next.js 14 App Router, TypeScript, Tailwind CSS, Framer Motion, and Groq SDK. The codebase already has Groq integrated in src/app/api/chat-companion/route.ts using llama-3.3-70b-versatile. The existing onboarding is in src/components/Onboarding.tsx and routes through src/app/page.tsx into src/app/dashboard/page.tsx via URL passport pattern. You are replacing the entire onboarding system. Study every existing file mentioned before writing a single line.

THE CORE PROBLEM YOU ARE SOLVING
The existing onboarding asks users to select from Student, Professional, Founder, Hobbyist as identity labels and then select Deep Dive, Project Based, Just Exploring as experience modes. These labels are meaningless for personalization. They get sent to Gemini as role context and produce faulty video recommendations because "Founder" tells Gemini nothing about what videos to show. Someone learning history cannot "build a project." Someone learning React for a job interview has nothing in common with someone exploring React out of curiosity even though both might call themselves a Student. The entire role and experience level system must be deleted and replaced with a context extraction system that captures what actually matters for learning personalization.

WHAT YOU ARE BUILDING
A conversational onboarding experience powered by Groq that feels like talking to a knowledgeable friend who is figuring out how to help you, not filling out a form. The conversation has one purpose — extract enough context about this specific learning situation to generate a genuinely appropriate first roadmap. The conversation should feel natural and end within three to five exchanges. It should never feel like an interrogation or a signup flow.
The conversation must silently extract six signals without ever asking about them directly as form fields. These six signals are the specific actionable goal behind why they want to learn this topic, their prior knowledge inferred from how they speak rather than self-reported, the domain type which is one of technical, academic, creative, or business, their time context meaning whether they have a deadline or are learning openly, their preferred session length inferred from how they describe their life situation, and their success definition meaning what done actually looks like for them personally.
When all six signals are captured the conversation ends naturally and transitions to roadmap generation. The user should feel like they just had a helpful conversation, not that they completed an intake form.

THE LEARNING CONTEXT OBJECT
This is the most important output of the entire phase. Everything downstream in the platform consumes this object. It must be stored in both localStorage for immediate use and Supabase for persistence across sessions. The object contains the topic as a specific string, the goal as a specific actionable description not a generic category, the domainType as exactly one of technical or academic or creative or business, the priorKnowledge as exactly one of none or adjacent or foundational or intermediate or advanced, the timeContext as either an object with a deadline date or the string open-ended, the sessionLength as exactly one of short meaning under 30 minutes or medium meaning 30 to 60 minutes or long meaning over 60 minutes, and the successDefinition as a specific string describing what completion looks like for this person. This object replaces every usage of role and experienceLevel throughout the entire codebase. Everywhere role or experienceLevel or userRole is passed to any API route or Gemini prompt, it must be replaced with relevant fields from this context object.

FILE ARCHITECTURE
You will create three new files and modify four existing files. Study all of them before beginning.
The first new file is a Groq API route specifically for onboarding conversation. This is separate from the existing chat-companion route because it has a completely different system prompt and a different job. Its job is not to answer questions but to conduct a structured extraction conversation while feeling unstructured. It must track conversation progress internally and signal when all six context signals have been captured. It returns both the AI message to display and a partial context object representing what has been extracted so far. When the context object is complete it returns a completion flag alongside the final context object.
The second new file is the new Onboarding component. This replaces src/components/Onboarding.tsx entirely. It renders as a chat interface not a step-by-step wizard. It has a message thread that grows as the conversation progresses. It has a text input at the bottom. It silently accumulates the context object from each API response. When the completion flag arrives it shows a brief confirmation moment then transitions to the roadmap. The visual design follows the platform aesthetic — deep navy background, teal accents, no gradients, clean sans-serif typography. The chat bubbles should feel light and conversational not like a customer service chat widget.
The third new file is a context utility that handles storing, retrieving, and validating the Learning Context Object in both localStorage and Supabase. It exports functions that any other part of the codebase can call to get the current user's learning context. This replaces the pattern of passing role and experience as URL params.
The four modified files are src/app/page.tsx which now renders the new Onboarding component, src/app/dashboard/page.tsx which must remove all references to role and experience URL params and instead read from the context utility, src/app/roadmap/page.tsx which must replace role and experience with context object fields when calling the generate-roadmap API, and src/app/api/generate-roadmap/route.ts which must replace the role and experienceLevel parameters with the richer context fields and update its Gemini prompt accordingly.

THE GROQ SYSTEM PROMPT DESIGN
This is the most critical piece of the entire phase. The system prompt for the onboarding Groq route must accomplish several things simultaneously. It must instruct the AI to conduct a natural conversation while extracting specific structured data. It must prevent the AI from asking multiple questions at once which feels like an interrogation. It must prevent the AI from being overly enthusiastic or using filler phrases like "Great question" or "Absolutely." It must instruct the AI to infer prior knowledge from vocabulary and context rather than asking "what is your experience level." It must handle the domain detection silently — if someone mentions Python and data pipelines it knows this is technical, if someone mentions understanding the French Revolution it knows this is academic. The system prompt must include explicit instructions about what signals have been captured in the current conversation so the AI knows when it has enough to end gracefully. The system prompt must also include the extraction schema so the AI returns structured JSON alongside its conversational message in every response.

THE CONVERSATION FLOW
The first message from the AI should be a single open question that invites the person to describe what they want to learn and why. Not "What is your role" and not "What topic do you want to learn." Something that naturally invites a fuller response like asking what they are trying to figure out or build or understand. From the first response the AI should already be extracting topic, early domain signals, and motivation hints.
The second exchange should naturally probe the gap between where they are and where they want to be. This surfaces prior knowledge without asking for it directly.
The third exchange should clarify what success looks like and implicitly surface time context. Someone who says they have an interview next week has told you their deadline. Someone who says they have been curious about this for a while has told you open-ended.
By the end of the third exchange ideally or the fifth exchange at absolute maximum the AI should have all six signals and should end the conversation with a natural closing statement that confirms what it understood and tells the user their personalized path is being built.

DOMAIN TYPE DETECTION LOGIC
The AI must classify domain type from the conversation content not from any explicit question. Technical means the user is learning a programming language, framework, tool, system design concept, mathematics, or any skill where there is objective right and wrong and where building something is a natural output. Academic means history, science, philosophy, literature, social sciences, or any domain where understanding and analysis are the primary outputs and there is no "project" to build. Creative means design, music, writing, photography, filmmaking, or any domain where subjective craft and personal expression are primary. Business means marketing, management, finance, entrepreneurship, or strategy where application to organizational or commercial goals is primary. This classification matters because it changes which scoring signals are used for video selection downstream. Technical content rewards GitHub links and longer duration. Academic content rewards narrative explanation and moderate duration. Creative content rewards demonstrated technique. Business content rewards case studies and practical frameworks.

PRIOR KNOWLEDGE INFERENCE RULES
Never ask "what is your experience level." Instead infer from signals in the conversation. None means the person describes the topic as something completely new to them, uses no domain vocabulary, or explicitly says they have never encountered it. Adjacent means they know related things but not this specifically, like knowing JavaScript but not React. Foundational means they understand core concepts but have not applied them deeply, indicated by using basic correct vocabulary but asking about fundamentals. Intermediate means they have worked with this before, indicated by using specific terminology correctly and asking about nuanced problems. Advanced means they are clearly experienced, indicated by asking about optimization, edge cases, architecture decisions, or comparing approaches.

CRITICAL CONSTRAINTS
The conversation must work for every possible topic without any topic-specific hardcoding. Someone learning Swahili and someone learning Kubernetes must both have a natural conversation that extracts the right context through the same conversation logic.
The completion detection must be robust. The API route should track which of the six signals have been captured after each exchange and only return the completion flag when all six are present with sufficient specificity. A vague goal like "I want to learn Python" is not sufficient — the conversation should continue to surface why specifically.
The transition from conversation to roadmap must feel intentional not abrupt. A brief moment where the UI acknowledges the conversation ended and shows what was understood before the roadmap generation begins.
All existing functionality must continue working. The URL passport pattern can be simplified but the roadmap page must still receive the context it needs. The localStorage key for the generated course must be cleared on new onboarding as it currently is.
The Supabase schema needs one new table for user learning contexts. The table stores user identifier if available or a session identifier if anonymous, the full context object as JSONB, the conversation transcript as JSONB for future analysis, and the timestamp. This table is the foundation for the study matching feature in a later phase.

WHAT NOT TO DO
Do not keep any reference to Student, Professional, Founder, or Hobbyist anywhere in the codebase after this implementation. Do not keep Deep Dive, Project Based, or Just Exploring as experience level options. Do not add new dropdown selectors of any kind. Do not make the conversation longer than five exchanges under any circumstances. Do not use loading spinners between messages — use a subtle typing indicator only. Do not animate the chat bubbles with entrance effects beyond a simple fade. Do not show the extracted context object to the user at any point during the conversation. Do not use the word "personalized" in the UI copy — show the personalization through what the roadmap contains not through claiming it is personalized.
